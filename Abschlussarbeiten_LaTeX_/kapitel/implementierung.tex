\section{Implementierung}

Dieses Kapitel beschreibt die technische Umsetzung des in der Konzeption entworfenen Bestellanforderungsprozesses. Ausgehend von der Projektstruktur werden die einzelnen Schichten der Anwendung -- Datenbankschicht, Service-Schicht, Workflow-Integration, Frontend und Deployment -- anhand von Codebeispielen erläutert. Die Implementierung erfolgt auf Basis des \gls{cap} mit Node.js, SAP Fiori Elements für die Benutzeroberfläche und \gls{spa} für den Genehmigungsworkflow. Die Bereitstellung wird als \gls{mta} auf Cloud Foundry realisiert.

\subsection{Datenbankschicht}

Das Domänenmodell wird deklarativ in \gls{cds} definiert. Auflistung~\ref{lst:schema} zeigt das vollständige Schema der Anwendung.

\begin{lstlisting}[language=Java,caption={Domänenmodell in CDS (schema.cds)},label={lst:schema}]
namespace purchase.approval;
using { cuid, managed, Currency } from '@sap/cds/common';
using { Attachments } from '@cap-js/attachments';

type RequisitionStatus : String(20) enum {
  Draft; Submitted; Approved; Rejected;
}
type ChangeSource : String(20) enum {
  USER; WORKFLOW; CALLBACK;
}

entity PurchaseRequisitions : cuid, managed {
  description       : String(256)  @mandatory;
  requester         : String(128)  @mandatory;
  costCenter        : String(10)   @mandatory;
  totalAmount       : Decimal(15,2);
  currency          : Currency;
  status            : RequisitionStatus default 'Draft';
  workflowId        : String(256);
  approverComment   : String(1000);
  approvedBy        : String(128);
  decidedAt         : Timestamp;
  statusCriticality : Integer default 0 @UI.Hidden;
  items         : Composition of many Items
                    on items.parent = $self;
  statusHistory : Composition of many StatusHistory
                    on statusHistory.parent = $self;
  attachments   : Composition of many Attachments;
}

entity Items : cuid {
  parent      : Association to PurchaseRequisitions;
  materialNo  : String(40)   @mandatory;
  description : String(256);
  quantity    : Decimal(13,3) @mandatory;
  unitPrice   : Decimal(15,2) @mandatory;
  currency    : Currency;
  netAmount   : Decimal(15,2);
}

entity StatusHistory : cuid, managed {
  parent     : Association to PurchaseRequisitions;
  oldStatus  : RequisitionStatus;
  newStatus  : RequisitionStatus;
  comment    : String(1000);
  changedBy  : String(128);
  source     : ChangeSource;
}
\end{lstlisting}

Die zentrale Entität \texttt{PurchaseRequisitions} nutzt zwei von \gls{cds} bereitgestellte Aspekte: Der Aspekt \texttt{cuid} generiert automatisch einen universell eindeutigen Bezeichner (UUID) als Primärschlüssel, sodass keine manuelle Schlüsselverwaltung erforderlich ist. Der Aspekt \texttt{managed} ergänzt die Felder \texttt{createdAt}, \texttt{createdBy}, \texttt{modifiedAt} und \texttt{modifiedBy}, die von der CAP-Runtime automatisch mit Zeitstempeln und Benutzerinformationen befüllt werden \citep{sap2024cap, herger2020cap}.

Der Typ \texttt{Currency} wird aus der mitgelieferten Bibliothek \texttt{@sap/cds/common} importiert. Dieser stellt einen assoziativen Verweis auf eine standardisierte Währungstabelle bereit, die ISO-4217-konforme Währungscodes enthält. Der Aufzählungstyp \texttt{RequisitionStatus} begrenzt die zulässigen Statuswerte auf die vier definierten Ausprägungen \texttt{Draft}, \texttt{Submitted}, \texttt{Approved} und \texttt{Rejected}. Der Aufzählungstyp \texttt{ChangeSource} unterscheidet drei Quellen für Statusänderungen: Manuelle Aktionen durch Benutzer (\texttt{USER}), automatische Aktionen durch den Workflow-Start (\texttt{WORKFLOW}) und Rückmeldungen von \gls{spa} (\texttt{CALLBACK}).

Von besonderer Bedeutung ist die Beziehung zwischen \texttt{PurchaseRequisitions} und \texttt{Items}. Die Verwendung von \texttt{Composition of many} statt einer einfachen Assoziation bewirkt, dass die Positionen im Lebenszyklus der übergeordneten Bestellanforderung verwaltet werden. Wird eine Bestellanforderung gelöscht, werden alle zugehörigen Positionen automatisch mit entfernt. Darüber hinaus ermöglicht die Komposition die gemeinsame Bearbeitung im Draft-Modus, da CAP zusammengesetzte Entitäten als eine transaktionale Einheit behandelt.

Die Entität \texttt{StatusHistory} protokolliert jeden Statuswechsel einschließlich des alten und neuen Status, des auslösenden Benutzers und der Quelle der Änderung. Durch die Komposition wird die Statushistorie gemeinsam mit der Bestellanforderung verwaltet und in der Benutzeroberfläche als eigene Sektion angezeigt.

Die Komposition \texttt{attachments} wird über das offizielle CAP-Plugin \texttt{@cap-js/attachments} bereitgestellt. Durch den Import der vorkonfigurierten \texttt{Attachments}-Entität und die Konfiguration in der Datei \texttt{package.json} (\texttt{kind:~db}) werden Dateianhänge als BLOB direkt in der HANA-Datenbank gespeichert. Das Plugin generiert automatisch die \gls{odata}-Medienressourcen für den Datei-Upload und -Download sowie die Fiori-Elements-Darstellung als Upload-Bereich in der Object Page.

Die Felder \texttt{approverComment}, \texttt{approvedBy} und \texttt{decidedAt} speichern die Genehmigungsinformationen, die über den Callback von \gls{spa} zurückgemeldet werden. Sie werden in der Benutzeroberfläche in einer eigenen Sektion \textit{Genehmigung} dargestellt.

Das Feld \texttt{statusCriticality} wird mit der Annotation \texttt{@UI.Hidden} versehen und ist somit in der Benutzeroberfläche nicht sichtbar. Es dient ausschließlich der farblichen Darstellung des Status in Fiori Elements, wobei ganzzahlige Werte den Farben Grau (0), Rot (1), Gelb (2) und Grün (3) entsprechen.

\subsection{Service-Schicht}

\subsubsection{Servicedefinition}

Die Servicedefinition exponiert das Domänenmodell als \gls{odata}-V4-Service und deklariert die anwendungsspezifischen Aktionen. Auflistung~\ref{lst:service-cds} zeigt einen Auszug der Servicedefinition.

\begin{lstlisting}[language=Java,caption={Servicedefinition in CDS (purchase-service.cds)},label={lst:service-cds}]
using { purchase.approval as db } from '../db/schema';

service PurchaseService
  @(path: '/purchase', requires: 'authenticated-user')
{
  @odata.draft.enabled
  entity PurchaseRequisitions
    as projection on db.PurchaseRequisitions actions {
      action submit() returns PurchaseRequisitions;
  };
  entity Items as projection on db.Items;

  @readonly
  entity StatusHistory
    as projection on db.StatusHistory;

  action workflowCallback (
    workflowId   : String,
    status       : String,
    comment      : String,
    approverName : String,
    decidedAt    : String
  ) returns PurchaseRequisitions;

  function getSpaInboxUrl() returns String;
}
\end{lstlisting}

Die Annotation \texttt{@odata.draft.enabled} aktiviert die Draft-Unterstützung für die Entität \texttt{PurchaseRequisitions}. Dies bewirkt, dass die CAP-Runtime automatisch eine Entwurfstabelle (\texttt{DraftAdministrativeData}) anlegt und die Schaltflächen \textit{Bearbeiten}, \textit{Sichern} und \textit{Verwerfen} in der Fiori-Elements-Oberfläche generiert werden. Benutzer können eine Bestellanforderung schrittweise bearbeiten, ohne dass Zwischenstände persistent in die aktive Tabelle geschrieben werden \citep{sap2024cap}.

Die gebundene Aktion \texttt{submit} wird auf Instanzebene definiert und dient der Einreichung einer einzelnen Bestellanforderung. Die Entität \texttt{StatusHistory} wird mit \texttt{@readonly} exponiert, da die Statushistorie ausschließlich serverseitig durch die Geschäftslogik befüllt wird und nicht vom Benutzer verändert werden darf. Die ungebundene Aktion \texttt{workflowCallback} stellt den Callback-Endpunkt dar, den \gls{spa} nach Abschluss einer Genehmigungsentscheidung aufruft. Neben der Workflow-ID und dem Entscheidungsstatus werden der Name des Genehmigers, ein optionaler Kommentar sowie der Entscheidungszeitpunkt übermittelt. Die Funktion \texttt{getSpaInboxUrl} liefert die konfigurierte URL des SPA My Inbox zurück.

Zusätzlich zur Servicedefinition enthält die CDS-Datei umfangreiche Fiori-Elements-Annotationen, welche die Benutzeroberfläche deklarativ konfigurieren. Auflistung~\ref{lst:annotations} zeigt die wesentlichen Annotationen.

Die Annotation \texttt{Capabilities} steuert die Bearbeitbarkeit der Bestellanforderung in Abhängigkeit vom Status. Über \texttt{UpdateRestrictions} und \texttt{DeleteRestrictions} werden die Schaltflächen \textit{Bearbeiten} und \textit{Löschen} ausgeblendet, solange sich die Anforderung im Status \texttt{Submitted} befindet, da in diesem Zustand die Genehmigungsentscheidung abgewartet werden muss. In allen anderen Status -- \texttt{Draft}, \texttt{Approved} und \texttt{Rejected} -- ist die Bearbeitung und Löschung möglich.

Die Annotation \texttt{UI.HeaderInfo} definiert den Titel und die Beschreibung der Object Page. \texttt{UI.LineItem} konfiguriert die Spalten der Tabellenansicht im List Report, wobei der Status über die \texttt{Criticality}-Eigenschaft farblich hervorgehoben wird. Die Annotation \texttt{UI.Identification} registriert die Schaltfläche \textit{Freigeben} als Aktion auf der Object Page. Über die Annotation \texttt{![@UI.Hidden]} wird diese Schaltfläche ausschließlich im Status \texttt{Submitted} ausgeblendet, sodass eine erneute Freigabe nach Genehmigung oder Ablehnung ohne vorherige manuelle Statusänderung möglich ist. Die \texttt{Facets}-Annotation definiert fünf Sektionen der Object Page: eine allgemeine Feldgruppe, die Genehmigungsinformationen (Entscheider, Zeitpunkt und Kommentar), die Tabelle der Positionen, den Statusverlauf als chronologische Historie aller Statusänderungen sowie den Anhängebereich für hochgeladene Dokumente. Die Sektion \textit{Anhänge} wird durch das \texttt{@cap-js/attachments}-Plugin automatisch als Upload-Bereich mit Drag-and-Drop-Unterstützung dargestellt.

\subsubsection{Geschäftslogik}

Die Event-Handler in \texttt{purchase-service.js} implementieren die serverseitige Geschäftslogik. Auflistung~\ref{lst:before-save} zeigt den \texttt{before SAVE}-Handler, der bei der Draft-Aktivierung die Beträge berechnet.

\begin{lstlisting}[language=Java,caption={Betragsberechnung und Status-Reset beim Speichern (before SAVE)},label={lst:before-save}]
this.before('SAVE', PurchaseRequisitions, async req => {
  const items = req.data.items || [];
  let total = 0;
  for (const item of items) {
    if (item.quantity != null && item.unitPrice != null) {
      item.netAmount = parseFloat(item.quantity)
                     * parseFloat(item.unitPrice);
    }
    total += parseFloat(item.netAmount || 0);
  }
  req.data.totalAmount = total;

  // Bei Bearbeitung einer nicht-Draft-Anforderung:
  // Status auf Draft zuruecksetzen
  if (req.data.ID) {
    const current = await SELECT.one
      .from(PurchaseRequisitions)
      .columns('status').where({ ID: req.data.ID });
    if (current && current.status !== 'Draft') {
      req.data.status = 'Draft';
      req.data.workflowId = null;
      req.data.approverComment = null;
      req.data.approvedBy = null;
      req.data.decidedAt = null;
    }
  }
});
\end{lstlisting}

Der \texttt{before SAVE}-Handler wird bei jeder Draft-Aktivierung ausgelöst. Er iteriert über alle Positionen, berechnet den jeweiligen Nettobetrag als Produkt aus Menge und Einzelpreis und summiert diese zum Gesamtbetrag der Bestellanforderung auf. Darüber hinaus prüft der Handler, ob eine bereits genehmigte oder abgelehnte Bestellanforderung bearbeitet wird. In diesem Fall wird der Status automatisch auf \texttt{Draft} zurückgesetzt und die vorherigen Genehmigungsinformationen werden entfernt. Dadurch kann der Anforderer die Anforderung erneut zur Genehmigung einreichen.

Auflistung~\ref{lst:submit} zeigt die zentrale \texttt{submit}-Aktion, die eine Bestellanforderung zur Genehmigung einreicht.

\begin{lstlisting}[language=Java,caption={Submit-Aktion mit Workflow-Start},label={lst:submit}]
this.on('submit', PurchaseRequisitions, async req => {
  const id = req.params[0]?.ID || req.params[0];
  const pr = await SELECT.one.from(PurchaseRequisitions)
    .where({ ID: id })
    .columns(pr => {
      pr('*'); pr.items(item => item('*'));
      pr.attachments(a => { a.ID; a.filename;
                             a.mimeType; });
    });

  if (!pr) return req.error(404, '...');
  if (pr.status === 'Submitted')
    return req.error(409, '...');
  if (!pr.items || pr.items.length === 0)
    return req.error(422, '...');

  await UPDATE(PurchaseRequisitions)
    .set({ status: 'Submitted' }).where({ ID: id });

  // Statushistorie: Draft -> Submitted
  await INSERT.into(StatusHistory).entries({
    parent_ID: id, oldStatus: 'Draft',
    newStatus: 'Submitted',
    comment: 'Zur Genehmigung eingereicht',
    changedBy: req.user?.id, source: 'USER'
  });

  try {
    const wfId = await workflowClient
      .startApprovalWorkflow(pr);
    await UPDATE(PurchaseRequisitions)
      .set({ workflowId: wfId }).where({ ID: id });
    // Statushistorie: Workflow gestartet
    await INSERT.into(StatusHistory).entries({
      parent_ID: id, oldStatus: 'Submitted',
      newStatus: 'Submitted',
      comment: 'Workflow gestartet (ID: ' + wfId + ')',
      changedBy: 'SYSTEM', source: 'WORKFLOW'
    });
  } catch (err) {
    await UPDATE(PurchaseRequisitions)
      .set({ status: 'Draft' }).where({ ID: id });
    // Statushistorie: Fehler-Rollback
    await INSERT.into(StatusHistory).entries({
      parent_ID: id, oldStatus: 'Submitted',
      newStatus: 'Draft',
      comment: 'Workflow-Start fehlgeschlagen',
      changedBy: 'SYSTEM', source: 'WORKFLOW'
    });
    return req.error(502, '...');
  }
  return SELECT.one.from(PurchaseRequisitions)
    .where({ ID: id });
});
\end{lstlisting}

Die \texttt{submit}-Aktion führt zunächst eine dreistufige Validierung durch: Es wird geprüft, ob die Bestellanforderung existiert, ob sie sich nicht bereits im Status \texttt{Submitted} befindet, und ob mindestens eine Position vorhanden ist. Durch diese Prüfung können auch bereits genehmigte oder abgelehnte Anforderungen direkt erneut eingereicht werden, ohne dass der Status zuvor manuell auf \texttt{Draft} zurückgesetzt werden muss. Beim Laden der Bestellanforderung werden neben den Positionen auch die Metadaten der Anhänge (ID, Dateiname und MIME-Typ) abgerufen, damit diese als Kontextdaten an den Workflow übergeben werden können. Anschließend wird der Status auf \texttt{Submitted} gesetzt und jede Statusänderung in der \texttt{StatusHistory} protokolliert. Der Genehmigungsworkflow in \gls{spa} wird gestartet und die zurückgegebene Workflow-Instanz-ID in der Entität gespeichert. Im Fehlerfall wird der Status auf \texttt{Draft} zurückgesetzt und auch der Fehler in der Statushistorie dokumentiert.

Auflistung~\ref{lst:callback} zeigt den Callback-Handler, der von \gls{spa} nach der Genehmigungsentscheidung aufgerufen wird.

\begin{lstlisting}[language=Java,caption={Workflow-Callback-Handler},label={lst:callback}]
this.on('workflowCallback', async req => {
  const { workflowId, status, comment,
          approverName, decidedAt } = req.data;
  const pr = await SELECT.one.from(PurchaseRequisitions)
    .where({ workflowId });

  if (!pr) return req.error(404, '...');
  if (pr.status !== 'Submitted')
    return req.error(409, '...');

  const newStatus = status === 'APPROVED'
    ? 'Approved' : 'Rejected';
  const now = decidedAt
    ? new Date(decidedAt) : new Date();

  const updateData = {
    status:          newStatus,
    approverComment: comment || null,
    approvedBy:      approverName || null,
    decidedAt:       now.toISOString()
  };
  // Bei Ablehnung: Workflow-ID zuruecksetzen
  if (newStatus === 'Rejected')
    updateData.workflowId = null;

  await UPDATE(PurchaseRequisitions)
    .set(updateData).where({ ID: pr.ID });

  // Statushistorie: Callback von SPA
  await INSERT.into(StatusHistory).entries({
    parent_ID: pr.ID, oldStatus: 'Submitted',
    newStatus: newStatus,
    comment: comment || newStatus + ' via SPA',
    changedBy: approverName || 'SPA-Workflow',
    source: 'CALLBACK'
  });

  return SELECT.one.from(PurchaseRequisitions)
    .where({ ID: pr.ID });
});
\end{lstlisting}

Der Callback-Handler empfängt die Workflow-ID, den Genehmigungsstatus, den Kommentar, den Namen des Genehmigers und den Entscheidungszeitpunkt von \gls{spa}. Über die gespeicherte \texttt{workflowId} wird die zugehörige Bestellanforderung identifiziert. Der Status wird entsprechend der Entscheidung auf \texttt{Approved} oder \texttt{Rejected} aktualisiert, und die Genehmigungsinformationen werden in den Feldern \texttt{approverComment}, \texttt{approvedBy} und \texttt{decidedAt} gespeichert. Bei einer Ablehnung wird zusätzlich die \texttt{workflowId} auf \texttt{null} zurückgesetzt, damit die Anforderung erneut eingereicht und einem neuen Workflow zugeordnet werden kann. Die Statusänderung wird in der \texttt{StatusHistory} mit der Quelle \texttt{CALLBACK} protokolliert. Diese Entkopplung über einen asynchronen Callback ermöglicht es, den Genehmigungsprozess vollständig in \gls{spa} abzubilden, während die CAP-Anwendung als System of Record den aktuellen Status und die Entscheidungsdokumentation verwaltet.

\subsection{Workflow-Integration}

Die Kommunikation mit der \gls{spa}-\gls{api} ist in der Klasse \texttt{WorkflowClient} in der Datei \texttt{srv/lib/workflow-client.js} gekapselt. Diese Klasse implementiert den OAuth2 Client Credentials Flow zur Authentifizierung und stellt Methoden für die Interaktion mit der Workflow-\gls{rest}-\gls{api} bereit.

Auflistung~\ref{lst:workflow-token} zeigt die Token-Beschaffung über den OAuth2 Client Credentials Flow.

\begin{lstlisting}[language=Java,caption={OAuth2-Token-Beschaffung (Auszug aus workflow-client.js)},label={lst:workflow-token}]
async _getToken() {
  const tokenUrl    = process.env.SPA_TOKEN_URL;
  const clientId    = process.env.SPA_CLIENT_ID;
  const clientSecret = process.env.SPA_CLIENT_SECRET;

  const response = await fetch(tokenUrl, {
    method: 'POST',
    headers: {
      'Content-Type':
        'application/x-www-form-urlencoded',
      'Authorization': 'Basic ' +
        Buffer.from(clientId + ':' + clientSecret)
          .toString('base64')
    },
    body: 'grant_type=client_credentials'
  });
  const data = await response.json();
  return data.access_token;
}
\end{lstlisting}

Die Methode \texttt{\_getToken} liest die Zugangsdaten aus Umgebungsvariablen und sendet einen HTTP-POST-Request an den Token-Endpoint der \gls{spa}-Instanz. Die Client-ID und das Client-Secret werden Base64-kodiert im \texttt{Authorization}-Header übertragen. Das zurückgegebene Zugriffstoken wird für nachfolgende \gls{api}-Aufrufe verwendet.

Auflistung~\ref{lst:workflow-start} zeigt die Methode zum Starten eines Genehmigungsworkflows.

\begin{lstlisting}[language=Java,caption={Start eines Genehmigungsworkflows},label={lst:workflow-start}]
async startApprovalWorkflow(requisition) {
  const items = requisition.items || [];
  const attachments = requisition.attachments || [];
  const appUrl = process.env.APP_URL || '';

  // Positionen als strukturierte Liste
  const itemsList = items.map(item => ({
    materialNo:  String(item.materialNo || ''),
    description: String(item.description || ''),
    quantity:    String(item.quantity || 0),
    unitPrice:   String(item.unitPrice || 0),
    netAmount:   String(item.netAmount || 0)
  }));

  // Anhaenge mit direkten Download-URLs
  const attachmentsList = attachments.map(a => ({
    filename: String(a.filename || ''),
    mimeType: String(a.mimeType || ''),
    url: appUrl ? appUrl
      + '/purchase/PurchaseRequisitions_attachments'
      + '(up__ID=' + requisition.ID
      + ',ID=' + a.ID
      + ',IsActiveEntity=true)/content' : ''
  }));

  // Deep-Link zur Fiori Object Page
  const detailLink = appUrl
    ? appUrl + '#/PurchaseRequisitions'
      + '(ID=' + requisition.ID
      + ',IsActiveEntity=true)' : '';

  const workflowContext = {
    requisitionId: requisition.ID,
    description:   requisition.description,
    requester:     requisition.requester,
    costCenter:    requisition.costCenter,
    totalAmount:   parseFloat(
                     requisition.totalAmount),
    items:          itemsList,
    attachments:    attachmentsList,
    attachmentCount: attachments.length,
    detailLink:     detailLink,
    callbackUrl:    process.env.CAP_CALLBACK_URL
  };
  // ... Retry-Mechanismus wie zuvor ...
}
\end{lstlisting}

Die Methode \texttt{startApprovalWorkflow} übergibt einen umfangreichen Kontext an die \gls{spa}-Workflow-\gls{api}. Neben den Kopfdaten der Bestellanforderung (ID, Beschreibung, Anforderer, Kostenstelle und Gesamtbetrag) werden die einzelnen Positionen als strukturierte Liste sowie die Metadaten der Anhänge einschließlich direkter Download-URLs übergeben. Die Download-URLs verweisen auf den \gls{odata}-Medien-Endpunkt der Anhänge, sodass der Genehmiger Dokumente direkt aus dem SPA-Genehmigungsformular herunterladen kann. Zusätzlich wird ein Deep Link zur Fiori-Object-Page der Bestellanforderung generiert, über den der Genehmiger bei Bedarf direkt zur Detailansicht navigieren kann.

Die Workflow-Definition-ID wird über eine Umgebungsvariable konfiguriert und verweist auf den im SPA-Designtime-Editor modellierten Genehmigungsprozess. Ein Retry-Mechanismus mit exponentiellem Backoff sorgt dafür, dass transiente Netzwerkfehler bis zu drei Mal automatisch abgefangen werden. Die Methode \texttt{\_apiRequest} kapselt die eigentliche HTTP-Kommunikation und setzt das zuvor beschaffte Zugriffstoken als Bearer-Token im \texttt{Authorization}-Header.

Die Konfiguration der Workflow-Integration erfolgt über die folgenden Umgebungsvariablen, die im Cloud-Foundry-Deployment als User-Provided-Variablen gesetzt werden:

\begin{itemize}
  \item \texttt{SPA\_API\_URL} -- Basis-URL der \gls{spa}-Workflow-\gls{api},
  \item \texttt{SPA\_TOKEN\_URL} -- Token-Endpoint für den OAuth2 Client Credentials Flow,
  \item \texttt{SPA\_CLIENT\_ID} und \texttt{SPA\_CLIENT\_SECRET} -- Zugangsdaten der Service-Instanz,
  \item \texttt{SPA\_WORKFLOW\_DEFINITION\_ID} -- ID der Workflow-Definition,
  \item \texttt{SPA\_INBOX\_URL} -- URL des SPA My Inbox für die Weiterleitung aus der Fiori-Oberfläche,
  \item \texttt{APP\_URL} -- Basis-URL der bereitgestellten Fiori-Anwendung für die Generierung von Deep Links und Download-URLs,
  \item \texttt{CAP\_CALLBACK\_URL} -- Callback-URL, die \gls{spa} nach der Genehmigungsentscheidung aufruft.
\end{itemize}

\subsection{Frontend}

Die Benutzeroberfläche basiert auf SAP Fiori Elements, einem Framework, das vollständige \gls{crud}-Oberflächen aus \gls{odata}-Metadaten und \gls{cds}-Annotationen generiert \citep{sap2024fiori}. Für die Hauptansichten -- den List Report und die Object Page -- ist kein benutzerdefinierter UI-Code erforderlich. Die gesamte Oberflächenstruktur wird durch die in Abschnitt~4.3.1 beschriebenen Annotationen deklarativ festgelegt.

Die Routing-Konfiguration der Anwendung wird in der Datei \texttt{manifest.json} definiert. Auflistung~\ref{lst:manifest} zeigt die relevante Routing-Konfiguration.

\begin{lstlisting}[language=Java,caption={Routing und Header-Aktion in manifest.json (Auszug)},label={lst:manifest}]
"PurchaseRequisitionsObjectPage": {
  "type": "Component",
  "name": "sap.fe.templates.ObjectPage",
  "options": {
    "settings": {
      "contextPath": "/PurchaseRequisitions",
      "content": {
        "header": {
          "actions": {
            "openSpaInbox": {
              "press": "purchase.approval.requisitions"
                + ".ext.ObjectPageExt.openSpaInbox",
              "text": "Genehmigung (SPA Inbox)",
              "visible":
                "{= ${status} === 'Submitted'}"
            }
          }
        }
      }
    }
  }
}
\end{lstlisting}

Die Object Page wird durch die Fiori-Elements-Vorlage \texttt{sap.fe.templates.ObjectPage} generiert. In der \texttt{header.actions}-Konfiguration wird eine benutzerdefinierte Schaltfläche \textit{Genehmigung (SPA Inbox)} registriert, die nur sichtbar ist, wenn der Status der Bestellanforderung \texttt{Submitted} lautet. Diese Schaltfläche verweist auf die Erweiterungsdatei \texttt{ObjectPageExt.js}, deren Implementierung in Auflistung~\ref{lst:ext} dargestellt ist.

\begin{lstlisting}[language=Java,caption={Erweiterung für die Object Page (ObjectPageExt.js)},label={lst:ext}]
sap.ui.define([], function () {
  "use strict";
  var SPA_INBOX_URL =
    "https://demo-ndbs.ap10.process-automation"
    + ".build.cloud.sap/comsapspaprocessautomation"
    + ".comsapspainbox/inbox.html";

  return {
    openSpaInbox: function () {
      sap.m.URLHelper.redirect(SPA_INBOX_URL, true);
    }
  };
});
\end{lstlisting}

Die Erweiterung definiert eine einzige Funktion \texttt{openSpaInbox}, die den Benutzer in einem neuen Browserfenster zur SPA My Inbox weiterleitet. In der My Inbox kann der Genehmiger die offenen Genehmigungsaufgaben einsehen und die Entscheidung treffen. Nach erfolgter Genehmigung oder Ablehnung löst \gls{spa} den in Abschnitt~4.3.2 beschriebenen Callback aus, der den Status in der CAP-Anwendung aktualisiert.

Durch die Aktivierung der Draft-Unterstützung mittels \texttt{@odata.draft.enabled} werden die Schaltflächen \textit{Bearbeiten}, \textit{Sichern} und \textit{Verwerfen} automatisch von Fiori Elements generiert. Der Benutzer kann eine Bestellanforderung über \textit{Bearbeiten} in den Draft-Modus versetzen, Positionen hinzufügen oder ändern und die Änderungen über \textit{Sichern} aktivieren. Erst durch den expliziten Aufruf der \textit{Freigeben}-Aktion wird der Genehmigungsprozess angestoßen.

\subsection{Deployment}

Die Bereitstellung der Anwendung auf der \gls{btp} erfolgt als \gls{mta} über Cloud Foundry. Der Build-Prozess wird mit dem Werkzeug \texttt{mbt build} ausgeführt, das auf Basis der Datei \texttt{mta.yaml} ein deploymentfähiges Archiv (\texttt{.mtar}) erzeugt. Die anschließende Bereitstellung erfolgt über den Befehl \texttt{cf deploy}. Auflistung~\ref{lst:mta} zeigt die Modulkonfiguration der \texttt{mta.yaml}.

\begin{lstlisting}[language=Java,caption={Modulkonfiguration der mta.yaml (Auszug)},label={lst:mta}]
modules:
  - name: purchase-approval-srv
    type: nodejs
    path: gen/srv
    parameters:
      buildpack: nodejs_buildpack
      memory: 256M
    requires:
      - name: purchase-approval-db
      - name: purchase-approval-auth
      - name: purchase-approval-destination

  - name: purchase-approval-db-deployer
    type: hdb
    path: gen/db
    requires:
      - name: purchase-approval-db

  - name: purchase-approval-app
    type: approuter.nodejs
    path: app/purchase-requisitions
    parameters:
      memory: 128M
    requires:
      - name: purchase-approval-auth
      - name: srv-api
\end{lstlisting}

Das Deployment umfasst drei Module:

\begin{enumerate}
  \item \textbf{purchase-approval-srv} -- Der CAP-Server als Node.js-Anwendung, der den \gls{odata}-Service bereitstellt und die Geschäftslogik ausführt. Dieses Modul benötigt Zugriff auf die HANA-Datenbank, den \gls{xsuaa}-Dienst für die Authentifizierung sowie den Destination-Dienst für externe Verbindungen.
  \item \textbf{purchase-approval-db-deployer} -- Der \gls{hdi}-Container-Deployer, der das CDS-Datenmodell in das HANA-Datenbankschema überführt. Er wird einmalig beim Deployment ausgeführt und beendet sich anschließend.
  \item \textbf{purchase-approval-app} -- Der Application Router (Approuter), der als Reverse Proxy die Fiori-Elements-Anwendung ausliefert und die Authentifizierung gegenüber dem \gls{xsuaa}-Dienst übernimmt.
\end{enumerate}

Diese Module greifen auf drei Cloud-Foundry-Ressourcen zu:

\begin{itemize}
  \item \textbf{purchase-approval-db} -- Ein \gls{hdi}-Container auf SAP HANA Cloud, der als isoliertes Datenbankschema für die Anwendung dient.
  \item \textbf{purchase-approval-auth} -- Eine \gls{xsuaa}-Service-Instanz, die auf Basis der Datei \texttt{xs-security.json} Rollen und Berechtigungen verwaltet. Die Sicherheitskonfiguration definiert die Scopes \texttt{Requester}, \texttt{Approver} und \texttt{PurchaseManager} sowie die zugehörigen Rollensammlungen.
  \item \textbf{purchase-approval-destination} -- Eine Destination-Service-Instanz, die für die Anbindung externer Dienste genutzt werden kann.
\end{itemize}

Die Konfiguration des \texttt{@cap-js/attachments}-Plugins erfolgt in der Datei \texttt{package.json} im Abschnitt \texttt{cds.requires}. Dort wird die Speicherart auf \texttt{db} gesetzt, sodass Dateianhänge als BLOB in der HANA-Datenbank abgelegt werden und kein externer Object Store erforderlich ist. Der im Plugin enthaltene Malware-Scanner wird im Prototyp durch eine Mock-Implementierung ersetzt.

Die Umgebungsvariablen für die \gls{spa}-Integration und die Anhänge-Integration (\texttt{SPA\_API\_URL}, \texttt{SPA\_TOKEN\_URL}, \texttt{SPA\_CLIENT\_ID}, \texttt{SPA\_CLIENT\_SECRET}, \texttt{SPA\_WORKFLOW\_DEFINITION\_ID}, \texttt{SPA\_INBOX\_URL}, \texttt{APP\_URL} und \texttt{CAP\_CALLBACK\_URL}) werden nach dem Deployment über den Befehl \texttt{cf set-env} am Modul \texttt{purchase-approval-srv} konfiguriert. Diese Trennung von Code und Konfiguration folgt dem Twelve-Factor-App-Prinzip und ermöglicht den Wechsel zwischen verschiedenen \gls{spa}-Instanzen ohne Codeänderung \citep{wiggins2017twelvefactor}.
